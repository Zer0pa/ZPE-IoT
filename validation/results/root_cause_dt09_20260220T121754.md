# DT-09 Root Cause Analysis (20260220T121754)

## Symptom
- Strict replay intermittently failed on `DT-09 Latency` even when all other mandatory DTs were green.
- Failure mode: `latency_mean_ms` exceeded 0.5 ms in isolated runs.

## Evidence
- Replay campaign `validation/results/strict_replay_campaign_20260220T114632.json`:
  - Run 1 failed (`DT-09 mean=0.652ms, p99=0.961ms`), runs 2-5 passed.
  - DT-10, DT-11, DT-22, and DT-27 remained PASS in all 5 runs.
- Final replay campaign `validation/results/strict_replay_campaign_20260220T121754.json`:
  - All 5 runs passed strict gates.

## Root Cause
- `DT-09` was sensitive to first-run host transients (CPU frequency ramp + allocator state), producing inflated native mean latency on cold starts.
- Prior stabilization (GC-controlled timed sampling + short warmup) reduced jitter but did not fully remove first-run drift.

## Corrective Changes
- File: `validation/destruct_tests/dt09_latency.py`
- Changes:
  - Added deterministic timed sampling harness with explicit warmup and GC control.
  - Kept latency thresholds unchanged (`mean < 0.5ms`, `p99 < 2.0ms`).
  - Increased warmup iterations from 256 to 2048 to eliminate cold-start drift.
  - Logged measurement metadata (`warmup_iterations`, `sample_count`, `inner_loops`) for provenance.

## Validation
- Final replay: `validation/results/strict_replay_summary_20260220T121754.md`
  - 5/5 strict runs green.
  - DT-09 values stayed within thresholds on every run.
- No regression detected in contamination/collision protections:
  - DT-22 PASS x5
  - DT-27 PASS x5
